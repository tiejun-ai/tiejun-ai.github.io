    Agent Skills: My Explanation and Thoughts

    June 29, 2026

    Agent Skills package a task’s prompt, tools, and resources into a reusable folder the agent loads on demand. Here’s my way of understanding why they exist — by tracing how we’ve fed knowledge to LLMs — and what makes them so important!

    Anthropic Engineering Blog: Equipping agents for the real world with Agent Skills

    1. The Road to Skills#

    Before Agent Skills showed up, the way we used LLMs went through a few stages of evolution. Each stage adds a better way to get the right information in front of the LLM.

    ① Vanilla ChatBot — the most primitive interaction, relying only on the model’s pretrained knowledge and the user’s prompt.

    User Prompt ─► LLM ─► answer

    ② Context-Augmented ChatBot / RAG (Retrieval-Augmented Generation) — to make up for the model’s missing external knowledge, we “feed” it more knowledge at inference time, usually obtained via a RAG retrieval process.

    User Prompt + resources (charts / web pages / local drafts / code) ─► LLM ─► answer

    ③ Tool Use / Function Calling — the AI is no longer just “talking”; we can feed the LLM with Tool Definitions to enable the LLM to interact with the outside world via Tool Use.

    Input:  User Prompt + resources + tool definitions
        (Python executor, MCP server, Bash terminal, ...)  ─► LLM

Run:    LLM enters the core Agent Loop
        think ─► act ─► observe ─► think ...
        dynamically calling tools until it returns an answer

    2. Problems: Poor Reuse; Error Prone#

    When you take these modes into complex or vertical-domain tasks, an obvious problem surfaces: poor reusability and error-prone contexts.

    Every time you ask an agent to solve the same kind of task — say “code review” or “writing a financial-report analysis” — the developer has to:

    1. Re-gather that scenario’s prompt, its reference resources (Assets), and its specific tool list (Tools).
    2. Cram all of that bulky information into the LLM’s context window.

    This is highly inefficient — the same engineering effort gets repeated again and again. It’s also error-prone: errors in context make token consumption spike, slow responses, and can even trigger the model’s “attention dispersion” — the more irrelevant material you stuff in, the higher the chance of hallucination.

            Task A ──► hand-assemble: Prompt + Assets + Tools ──► huge context
        Task A again ──► assemble it ALL over again ──► huge context
        Task B ──► hand-assemble again ──► huge context
                          ▲
              repetitive, bloated, error-prone

    Just as subroutines (functions) and classes were invented to structure code and enable reuse, we need a way to structure knowledge — prompts, tools, and assets — so it can be reused too.

    3. The Solution: Skills as Dynamically Loaded Knowledge#

    Agent Skills package all the knowledge a specific task needs into one standardized, pluggable module:

    Skill = Name + Description + Prompts + Tools + Assets
         │       │            │          │       │
       what    when to      specialized  task-   reference
       it is   trigger it   instructions specific files
                                          tools

    The workflow changes. With Skills, you no longer hand-assemble a long context every time. The system automatically finds the relevant Skill based on the actual context, and pulls it into the conversation only when needed.

            User task ─► agent scans Skill descriptions
                        │
                        ▼
                 relevant Skill? ──► load it into context on demand
                        │
                        ▼
                 proceed with the right prompt + tools + assets

    4. What a Skill Actually Is#

    A Skill is just a folder of organized files — instructions, scripts, and resources — that an agent can discover to perform a specific task. At its root sits a SKILL.md file.

    pdf/
├── SKILL.md          # frontmatter + core instructions
├── forms.md          # form-specific guidance
├── reference.md      # general PDF operations
└── scripts/
    ├── check_fillable_fields.py
    ├── convert_pdf_to_images.py
    ├── extract_form_field_info.py
    └── fill_pdf_form_with_annotations.py

    5. The Mechanism: Progressive Disclosure#

    How can we load skills without blowing up the context window? Progressive disclosure — load information in tiers, only as deep as the task requires. Like a manual: read the table of contents first, then jump to the one chapter you need. Typically a Skill has the following three levels of knowledge:

    To see this play out, follow what enters the context window when a user asks Claude to fill out a PDF form:

    Source: Anthropic — Equipping agents for the real world with Agent Skills

    1. Start: the context holds the agent’s system prompt with a name and description from each installed Skill (bigquery, docx, nda-review, pdf, …), plus the user’s message: “Fill out this PDF based on what you know about me.”
    2. Trigger: Claude recognizes the pdf skill is relevant and reads it — Bash("cat /mnt/skills/pdf/SKILL.md") — pulling the Level 2 body into context.
    3. Drill down: the SKILL.md points to a more specific file, so Claude reads forms.md — a Level 3 file — only now that the task calls for it.
    4. Proceed: with just the right instructions loaded from forms.md, Claude completes the task.

    Note that Skills irrelevant to the task — like bigquery and docx — never load beyond their Level 1 metadata (name and description); the rest of those Skills stays out of context entirely.

    6. Skills vs Tools#

    Skills and tools are easy to confuse, but they help an AI Agent in different ways:

    Tools provide the agent with capabilities to act. Skills extend the agent with expertise — the on-demand knowledge to finish a task; a Skill often includes a set of prompts, resources, and in particular, tool definitions, as a knowledge package.

    Key Takeaways#

    • The road to Skills: vanilla chatbot → RAG → tool use → Skills. Each step delivers the right context, only when it’s needed.
    • The problem: recurring vertical tasks force you to re-assemble prompt + tools + assets every time, bloating context and inviting hallucination.
    • The idea: Skill = Name + Description + Prompts + Tools + Assets — a pluggable folder the agent discovers and loads on demand.
    • Why it scales: progressive disclosure loads metadata → instructions → detail in tiers, so only relevant knowledge is loaded.
    • Skills vs tools: tool definitions enable agents to act; Skills extend the agent with knowledge.

    In one line: Agent Skills are composable, file-based expertise — and progressive disclosure is what lets an agent carry unlimited knowledge while paying only for what it uses.

    Further Thoughts#

    • Skills are an important step toward structuring and reusing external knowledge, just as subroutines (functions) and classes structure code. Because AI Agents are driven by the knowledge missing from an LLM’s pre-training, Skills are a critical component of AI Agent engineering. Given the same LLM, an AI Agent is only as powerful as the Skills and Tools provided to it. Any company, organization, or an individual can benefit from developing effective Skills to enable better AI applications.
    • Since a Skill is just a package of knowledge, the industry should develop powerful AI tools to help create — or even automatically generate and maintain — Skills. A particularly interesting problem is how to derive Skills automatically or semi-automatically from existing knowledge in various forms — wikis, books, documents, code, and people’s conversations with chatbots. At a high level, RAG is one early attempt at this. After all, if AI can help us write code, it can definitely help us write Skills. The future is exciting!
