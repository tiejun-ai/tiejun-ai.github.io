How Does an LLM Know Tool Use?

June 25, 2026

The power of AI Agents comes from an LLM’s ability to call Tools. Where does this magic come from? Here is a high-level overview of it.

TL;DR#

A common misconception is that an LLM directly reaches out to the internet, runs code, or boots up an application when it uses a tool. In reality, the LLM never executes the tool. It is still just a text predictor.

Another misconception is that any LLM will reliably emit a well-formed tool call (as JSON) on its own, just by being asked. A capable model can be coaxed into it through prompting alone — but doing it reliably (right tool, right arguments, valid format, right moment) is what special training buys you. That training is exactly what turned tool use from a fragile prompting trick into a native, dependable feature.

Actually, “Tool use” (a.k.a. function calling) is a carefully orchestrated dance of three parts:

  1. A structured prompt — the AI Agent (e.g. a chatbot, or Claude Code) hands the LLM a list of available tools (as a JSON schema) inside the system prompt.
  2. Special training — the LLM has been fine-tuned to emit a structured tool-call request when it recognizes the need, instead of answering in prose.
  3. An external runtime — the AI Agent watches the model’s output, executes the tool function call for it, and feeds the result back into the conversation with the LLM.

The model decides what to call and with what arguments. The surrounding software in the AI Agent does the actual calling.

Problem#

LLMs are frozen text predictors. Their knowledge is static, they can’t do reliable arithmetic, and they can’t see anything live — today’s weather, your database, a stock price. Asking the raw model “What’s the weather in Paris right now?” can only ever produce a plausible-sounding guess.

Tool use bridges that gap. But the bridge is built by teaching the model to ask for an action in a format that external software (AI Agent) can recognize and carry out. The model produces a function call request; the AI Agent runtime runs it.

Method Explained#

End to end, a single tool-use turn looks like this:

            ┌──────────────────────────── conversation history ────────────────────────────┐
 SYSTEM     │ "You have tools:                                                              │
 PROMPT     │   get_current_weather(location: str, unit: 'celsius'|'fahrenheit')            │
            │    → Get current weather data for a city"                                     │
            └──────────────────────────────────────────────────────────────────────────────┘
 USER  ──►  "What's the weather like in Paris right now?"
              │
              ▼
        ┌───────────┐   (1) recognizes static knowledge can't answer; matches the request
        │    LLM    │       to the weather-tool description, and instead of prose emits:
        └───────────┘
              │
              ▼  (2) structured tool-call request
        { "name": "get_current_weather",
          "arguments": { "location": "Paris", "unit": "celsius" } }
              │
              ▼  (3) backend RUNTIME sees the call, PAUSES the model,
        ┌───────────┐      makes the real API call to a weather service…
        │  RUNTIME  │ ───► 🌐 weather API ───► { "temp_c": 22, "sky": "sunny" }
        └───────────┘      …and pipes the result back as a `tool_result` message
              │
              ▼  (4) model UNPAUSES, reads the fresh data, writes prose
        ┌───────────┐
        │    LLM    │ ───► "It's currently 22°C and sunny in Paris!"
        └───────────┘

Walking through the four moments:

1. The Blueprint (the system prompt)#

Before you type anything, the AI Agent injects a hidden system prompt listing the available tools. Each tool is described with a strict blueprint — usually a JSON schema — that spells out:

  • What the tool does: e.g. "Get current weather data for a city".
  • What inputs it requires: e.g. a location string and an optional unit enum (celsius / fahrenheit).

Because the model is highly sensitive to semantic meaning, it reads these descriptions and understands conceptually what each tool is for.

Concretely, with the OpenAI API you pass the tool blueprint in the tools parameter of the request:

from openai import OpenAI
client = OpenAI()

tools = [{
    "type": "function",
    "function": {
        "name": "get_current_weather",
        "description": "Get current weather data for a city",
        "parameters": {
            "type": "object",
            "properties": {
                "location": {"type": "string", "description": "City name, e.g. 'Paris'"},
                "unit": {"type": "string", "enum": ["celsius", "fahrenheit"]},
            },
            "required": ["location"],
        },
    },
}]

response = client.chat.completions.create(
    model="gpt-4o",
    messages=[{"role": "user", "content": "What's the weather like in Paris right now?"}],
    tools=tools,
)

# The model doesn't answer in prose — it returns a tool call to run:
call = response.choices[0].message.tool_calls[0]
print(call.function.name)       # "get_current_weather"
print(call.function.arguments)  # '{"location": "Paris", "unit": "celsius"}'

2. The Trigger (recognizing the need)#

When you ask about Paris weather, the model realizes its static training data can’t answer, matches your request against the tool descriptions, and decides to act. Instead of replying in English, it switches gears and outputs a tool-use request (often wrapped in special syntax like <tool_call> or raw JSON) declaring its intent:

{
  "name": "get_current_weather",
  "arguments": { "location": "Paris", "unit": "celsius" }
}

3. The Handshake (the external environment)#

The code running the model — the AI Agent — constantly watches the output stream. The moment it sees that tool-call JSON from the LLM, it takes the generated arguments ("Paris", "celsius"), makes the real API call to a weather service, grabs the live data, and pipes it back into the conversation history as a new message labeled tool_result.

This is the whole trick: the model only ever produced text describing a function call. Software in the AI Agent did the calling.

4. The Response (synthesizing the answer)#

With the tool call result now sitting in its context, the LLM generates a reply: “It’s currently 22°C and sunny in Paris!”

How Do LLMs Learn This Behavior?#

An LLM isn’t born knowing how to write JSON tool call requests. It’s trained into the behavior:

  • Supervised Fine-Tuning (SFT): the model is trained on thousands of mock transcripts shaped like [user asks] → [model emits a JSON tool call] → [system inserts a tool result] → [model gives the final answer]. It learns to reproduce this pattern.
  • Reinforcement Learning (RLHF / DPO): the model is rewarded for generating valid arguments that don’t crash the runtime, and penalized for hallucinating inputs or picking the wrong tool.

How the big labs actually train it#

The flagship labs’ technical reports reveal the high-level recipes:

OpenAI — API-first#

Native “Function Calling” shipped in June 2023 (gpt-3.5-turbo-0613). They fine-tuned on data where system prompts contained API schemas and target outputs were stringified JSON. Later models (GPT-4o) were tuned for parallel function calling.

Anthropic — agentic feedback loops#

Claude is trained to treat tool descriptions as hard constraints, using RLHF and DPO to penalize hallucinated or unlisted arguments. More recently, programmatic tool calling trains the model to emit orchestration code (Python loops) that runs in a sandbox — processing thousands of tokens of intermediate data and returning only the compressed final answer to the main context.

Google — architectural integration#

The Gemini technical reports note that tool use isn’t bolted on in post-training — structured API interactions, database queries, and multi-turn coding logs are blended into pre-training alongside regular text. Automated verification loops penalize any tool argument that fails a JSON validator or compiler, which is why Gemini leans on highly accurate grammar-constrained sampling.

The Research & Engineering Landscape#

Tool use evolved from a novel idea into standard AI-engineering infrastructure. A few landmarks:

Foundational research

  • Toolformer (Schick et al., 2023) — the seminal proof that a model can teach itself, via self-supervision, when to emit an API call, what arguments to pass, and how to fold the result back into next-token prediction without losing its text-generation ability.
  • Gorilla (Patil et al., 2023) — scaled the idea from a handful of tools (calculator, calendar) to thousands of complex, ever-changing web APIs, using instruction tuning plus a retriever to mitigate API-documentation hallucination.

Engineering standards

Moving from research to production exposed severe fragmentation: every provider wanted a different schema, every data source needed bespoke glue code. The industry’s answer is unified protocol layers — most prominently the Model Context Protocol (MCP) from Anthropic. MCP standardizes how an AI Agent talks to an “MCP server” (a tool or data-source wrapper). Engineers deploy modular MCP servers; the LLM dynamically discovers the available tools and data source from MCP servers.
