    AI Search Architecture: A Single-Diagram Summary

    July 6, 2026

    Modern AI Search — the retrieval engine behind RAG, AI chatbots, and answer engines — collapses into one pipeline. Here it is in a single diagram, with a concise walk through each stage and why it’s there.

    This post is the conceptual map. If you want to see these ideas in running code, I’ve built AI Search step by step in companion posts:

    • Build AI Web Search Demo with Claude Code — a minimal RAG pipeline: search the web, pass results to the LLM, synthesize a cited answer.
    • Build AI Web Search Demo V2 (Agentic Search) — an agentic loop where the LLM decides what to search, when to search again, and when it has enough to answer.

    The Diagram#

    I like to simplify a modern AI Search architecture into this one pipeline:

    User Query
   ↓
Query Understanding
   – rewrite / expand / classify / route
   ↓
Candidate Retrieval
   – BM25 / sparse
   – dense vector
   – metadata filter
   – hybrid fusion / RRF
   ↓
Reranking
   – cross-encoder
   – LLM reranker
   – business rules / freshness / authority
   ↓
Context Construction
   – dedup
   – diversify
   – compress
   – citation mapping
   ↓
LLM Answer / Search Result UI
   ↓
Evaluation & Observability
   – recall@k / nDCG
   – answer faithfulness
   – latency / cost
   – user feedback

    The whole thing is a funnel: each stage takes a broad, cheap, noisy set of candidates and makes it narrower, more expensive, and more precise — until a handful of high-quality passages reach the LLM.

    1. Query Understanding#

    The raw user query is rarely the best thing to search with. Before touching any index, reshape it:

    • rewrite — fix a messy query, resolve pronouns, fold in conversation history (“what about its revenue?” → “what about Meta’s revenue?”).
    • expand — add synonyms or generate sub-queries so recall doesn’t hinge on the user’s exact wording.
    • classify — figure out intent: navigational vs. informational, needs fresh data vs. static knowledge.
    • route — send the query to the right backend: a vector index, a SQL table, a web-search API, or straight to the LLM if no retrieval is needed at all.

    This stage is cheap but high-leverage — a bad query wastes every stage after it.

    2. Candidate Retrieval#

    Now cast a wide net. The goal here is recall: don’t miss anything relevant. Speed matters more than precision because we’re scanning the whole corpus.

    • BM25 / sparse — classic keyword search. Unbeatable for exact matches: error codes, model numbers, names, code symbols.
    • dense vector — embedding search for semantic similarity, catching paraphrases keyword search misses.
    • metadata filter — hard constraints: date range, language, permissions, tenant, source.
    • hybrid fusion / RRF — BM25 and dense search each return a ranked list; Reciprocal Rank Fusion (RRF) merges them into one, using ranks rather than raw scores so no normalization is needed.

    Keyword and vector search cover each other’s blind spots — hybrid is the default for a reason.

    3. Reranking#

    Retrieval optimizes for “don’t miss anything.” Reranking optimizes for “order it precisely.” It re-scores a small candidate set (say, top-150) with a heavier, sharper model:

    • cross-encoder — reads the query and each candidate together (not as separate embeddings), so it models relevance far more accurately than a dot product.
    • LLM reranker — use an LLM to score or order candidates; most flexible, most expensive.
    • business rules / freshness / authority — the part pure relevance models ignore: boost recent content, trust authoritative sources, apply domain-specific ranking logic.

    Two-stage retrieval — fast broad recall, then slow precise reranking — is the workhorse pattern of modern search.

    4. Context Construction#

    The reranked list still isn’t ready for the LLM. Assemble it into a clean, compact context:

    • dedup — drop near-duplicate passages that waste tokens and skew the answer.
    • diversify — cover different facets of the query instead of ten paraphrases of the same fact.
    • compress — trim or summarize passages to fit the context window and cut cost.
    • citation mapping — track which source each snippet came from, so the final answer can cite it.

    This is the quiet stage that decides answer quality — garbage-in still means garbage-out, no matter how good retrieval was.

    5. LLM Answer / Search Result UI#

    The payoff. Either the LLM generates a grounded answer from the constructed context (RAG / answer engine), or the ranked results are rendered directly in a search UI. Citations from the previous stage let the user verify claims and click through to sources.

    6. Evaluation & Observability#

    Everything above is a system you have to keep working, so this stage wraps around the whole pipeline:

    • recall@k / nDCG — is retrieval finding the right documents, and ranking them well?
    • answer faithfulness — is the LLM’s answer actually grounded in the retrieved context, or hallucinating?
    • latency / cost — the operational budget that constrains every stage above.
    • user feedback — clicks, thumbs, and follow-ups — the ground-truth signal that feeds back into every earlier stage.

    Without this stage you can’t tell which of the five stages above is failing.

    My Thoughts#

    Every stage is a recall-vs-precision trade-off. Retrieval maximizes recall cheaply; reranking maximizes precision expensively; context construction squeezes out the noise. Read top-to-bottom, the pipeline is just a funnel that spends more compute per candidate the fewer candidates remain.

    The stages are stable; the implementations aren’t. BM25 has been around for decades; cross-encoders and LLM rerankers are recent; context construction is being reshaped by cheap LLM inference. But the shape — understand, retrieve, rerank, construct, answer, measure — has held across web search, RAG, and today’s answer engines.

    Big picture. As LLM inference gets cheaper, more of this pipeline gets absorbed into the model: query understanding becomes an LLM rewrite, reranking becomes an LLM reranker, and agentic systems like Claude Code sometimes replace the whole retrieval stack with a plain grep. The boundaries between the stages will keep shifting — but knowing the stages is what lets you reason about where to spend your next dollar of compute.

    Key Takeaways#

    • Query Understanding — reshape the query (rewrite / expand / classify / route) before searching; cheap but high-leverage.
    • Candidate Retrieval — cast a wide net for recall; hybrid (BM25 + dense vector) fused with RRF is the default.
    • Reranking — a heavier, sharper second stage (cross-encoder / LLM) that trades speed for precision, plus business rules.
    • Context Construction — dedup, diversify, compress, and map citations to turn a ranked list into a clean LLM context.
    • LLM Answer / UI — generate a grounded, cited answer or render the results.
    • Evaluation & Observability — recall@k / nDCG, faithfulness, latency / cost, and user feedback close the loop.
    • Core lesson: AI Search is a funnel — each stage spends more compute per candidate as the candidate set shrinks.
