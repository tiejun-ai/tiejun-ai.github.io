Anthropic Deep Research: Lessons from a Canonical Multi-Agent AI Application

June 26, 2026

Anthropic published an excellent engineering blog on how they built their Deep Research feature on a multi-agent architecture. It’s a canonical multi-agent system, full of valuable innovations and hard-won practical lessons that are well worth studying as an AI engineer.

Original: How we built our multi-agent research system

1. Why Multi-Agent?#

Anthropic’s definition of Deep Research: open-ended problems where you can’t predict the steps in advance. You can’t hardcode a fixed path for exploring a complex topic, because the process itself is dynamic and path-dependent — as the investigation unfolds, the agent needs to pivot, chase down subtopics, or search for new information — just like a human doing deep research.

Thus a multi-agent architecture is a good fit for Deep Research:

  1. Parallel exploration — multiple subagents run at the same time, each with its own independent context window, exploring different facets of the problem.
  2. Separation of concerns — each subagent has its own tools, prompt, and exploration trajectory. This reduces path dependence and lets each line of investigation go deeper and stay more independent.
  3. Breaking the single-context ceiling — research often needs to process more information than a single agent’s context window can hold; spreading it across multiple agents fits far more in total context.

How well does it work? A multi-agent system using Claude Opus 4 model as the lead agent and Claude Sonnet 4 model as subagents outperformed a single-agent system (using Opus 4) by 90.2% on Anthropic’s internal research eval.

A classic example: find every board member of every IT company in the S&P 500. This kind of breadth-first task — fanning out many searches at once — fails for a single agent searching sequentially, but multiple subagents dividing the work in parallel can handle it.

2. Architecture: The Orchestrator-Worker Pattern#

The core is an orchestrator-worker pattern:

  • Lead Agent — analyzes the problem, forms a strategy, writes its plan to external memory (so a context overflow doesn’t wipe out the strategy), then spawns several subagents.
  • Subagents — each gets a specific task, searches independently, uses interleaved thinking to evaluate results, and returns only its filtered findings to the lead. A subagent acts like an “intelligent filter.”
  • CitationAgent — at the very end, dedicated to attributing each conclusion back to its sources.

The key is parallelism: the Lead Agent spawns 3–5 subagents at once, and each subagent fires multiple tool calls in parallel. For complex problems, this can cut research time by up to 90%.

3. The Cost of Parallelism: Token Economics#

There’s no free lunch. Multi-agents are powerful and parallelism is fast, but they burn tokens:

   regular chat          ▏ 1×    (baseline)
   single agent          ▏▏▏▏ ~4×
   multi-agent system    ▏▏▏▏▏▏▏▏▏▏▏▏▏▏▏ ~15×

Hence an important principle: multi-agent is only worth it when the task’s value is high enough to cover this cost.

Why does multi-agent win? Anthropic offers an interesting and important conclusion:

Multi-agent systems work mainly because they help spend enough tokens to solve the problem.

On the BrowseComp eval, three factors together — token usage, number of tool calls, and model choice — explained 95% of the performance variance, and token usage alone explained 80%. In other words, “feeding” a task enough tokens to explore is itself the main source of performance — and multi-agent is precisely a way to scale token usage beyond what a single agent’s context can hold.

4. Prompt Engineering: The Primary Lever to Harness Agents#

Every agent uses an LLM driven by a prompt, so as Anthropic puts it: “prompt engineering was our primary lever for improving these behaviors.” Anthropic distilled 8 principles:

  1. Think like your agents. Step through the agent’s behavior to see its failure modes (spawning too many subagents, verbose queries, picking the wrong tool).
  2. Teach the orchestrator how to delegate. Each delegated task must spell out the objective, output format, which tools to use, and where the boundaries are.
  3. Scale effort to query complexity. Put the rules in the prompt: simple fact research = 1 agent / 3–10 calls; comparison research = 2–4 subagents; complex research = 10+ subagents with clear divisions.
  4. Tool design and selection are critical. Treat tool interfaces as UX. An agent that picks the wrong tool almost always fails; each tool needs a clear, non-overlapping description.
  5. Let agents improve themselves. Claude 4 can diagnose its own failures and improve prompts. A “tool-testing agent” that rewrote tool descriptions cut later agents’ task completion time by 40%.
  6. Start wide, then narrow down. Begin with short, broad queries to see what’s out there, then progressively focus — mirroring how human experts research.
  7. Guide the thinking process. Use extended thinking as a “controllable scratchpad.” The Lead Agent uses it to plan; subagents use interleaved thinking after each tool call to assess quality, find gaps, and refine the next step.
  8. Parallel tool calling transforms speed. The Lead Agent spawns subagents in parallel, and subagents call tools in parallel, compressing complex research time by up to 90%.

5. Evaluation#

  • Start small. About 20 real-scenario queries is enough; early iterations with help of the eval have outsized impact (success rates can jump from 30% to 80%).
  • LLM-as-judge. Use a single prompt to score 0.0–1.0 across multiple dimensions: factual accuracy, citation accuracy, completeness, source quality, and tool efficiency.
  • Human evaluation is indispensable. People catch problems the LLM judge misses — like agents favoring SEO content farms over authoritative academic PDFs.

6. Production Challenges#

Putting agents into production surfaces problems ordinary software doesn’t have:

  • State and error accumulation. Agents run for a long time while holding state, so a small error compounds. Fix: durable execution, checkpoint-and-resume, and letting agents handle tool failures gracefully.
  • Non-deterministic debugging. The same prompt can produce different decisions each run. Fix: full production tracing and monitoring of decision patterns.
  • Deployment coordination. Agents may be mid-run when you update. Fix: rainbow deployments — old and new versions live at once, gradually shifting traffic.

Key Takeaways#

  • Why: Deep Research tasks are unpredictable, the information exceeds a single agent’s context, and the work is parallelizable → multi-agent.
  • Architecture: orchestrator-worker — Lead Agent delegates + subagents run in parallel + CitationAgent attributes sources.
  • Results: multi-agent systems score 90.2% higher than a single agent; up to 90% faster; but ~15× the tokens of a chat; token usage alone explains 80% of the performance variance (three factors together, 95%).
  • Core conclusion: multi-agent works mainly because it can “spend enough tokens to solve the problem.”
  • Prompt: prompt engineering is the primary lever — delegate specifically, scale effort to complexity, treat tool design as UX, start wide then narrow, let agents improve themselves, and exploit parallelism.

In one line: multi-agent suits high-value, parallelizable tasks whose information exceeds a single context — but it relies on careful prompt engineering, good tool design, solid evaluation, and a production system that can survive long-running execution.
