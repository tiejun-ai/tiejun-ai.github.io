OneRec: Unifying Retrieve and Rank with Generative Recommender and Preference Alignment — My Explanation and Thoughts

July 13, 2026

Understanding OneRec (Kuaishou) — a single generative recommender model that replaces the whole stack of retrieve→rank recommendation models, generates a full session of videos at once, and is aligned with user preferences via a reward-model-driven self-improving DPO loop.

Paper: OneRec: Unifying Retrieve and Rank with Generative Recommender and Preference Alignment (Deng et al., Kuaishou, 2025)

TL;DR#

Modern recommenders run a multi-stage cascade — retrieval → pre-ranking → ranking — where each stage is a separate model and each one’s quality caps the next. Recent generative-retrieval work like TIGER (see my post) structures retrieval as an LLM-style Next Semantic ID prediction problem — but it only replaces the retrieval stage.

OneRec unifies the entire recommendation pipeline into one encoder–decoder Transformer. It reads a user’s watch history and generates the next session of videos directly — retrieve and rank become a single autoregressive decode. Three ideas make it work: (1) a sparse Mixture-of-Experts decoder so the model can be scaled cheaply; (2) session-wise generation — produce a whole list of ~5–10 videos at once instead of one item at a time; (3) Iterative Preference Alignment (IPA) — a reward model + DPO loop that teaches the model which sessions users actually enjoy. Deployed on Kuaishou’s main feed (hundreds of millions of DAU), it lifted total watch-time by 1.68% in A/B tests — a large win for Generative Recommendation at industry scale.

It’s also an exciting attempt to replace the entire multi-stage recommendation cascade with a single model trained end-to-end — reflecting a broader industry trend of giving ML models more freedom (less hand-engineered pipeline structure) to unlock higher performance.

Problem#

The cascade. To recommend efficiently out of a corpus of ~10¹⁰ videos, industrial recommenders funnel candidates through three stages, each heavier and more accurate than the last — retrieval (~10⁵ candidates) → coarse ranking (~10³) → fine ranking (~10²). OneRec replaces all of it with one encoder→decoder pass (Figure 1a vs 1b):

OneRec’s unified generation (a) vs the traditional multi-stage cascade (b). Figure 1 from the paper.

Two structural problems with the cascade:

  • Each stage is an upper bound on the next. Ranking can only reorder what retrieval passed up; anything dropped early is gone forever. The stages are trained independently, so errors compound.
  • Generative Recommendation is limited to retrieval. Methods like TIGER reframed retrieval as generating item IDs — but they still fed a downstream ranking cascade.

OneRec’s attempt: replace the whole cascade with one generative model. Given a user’s history, directly generate the final session of videos — no separate retrieval, pre-ranking, or ranking models. Retrieve and rank become the same autoregressive decode.

Method Explained#

OneRec has several components: item tokenization (turn each video into a short tuple of semantic ID tokens); a generative model (read user history, decode the next session); and an iterative DPO preference-alignment stage (IPA) that fine-tunes it with a reward model capturing what users actually like.

Step 1 — Tokenize each video into a Semantic ID (balanced K-means)#

Like TIGER, OneRec gives every item a Semantic ID: a short tuple of codewords where similar items share prefixes. Each video starts as a multi-modal embedding e, which is quantized into L = 3 tokens (one per level).

But the quantizer differs. TIGER trained and used an RQ-VAE model; OneRec found that RQ-VAE produces a badly unbalanced code distribution — a few codes absorb most items while most codes are nearly empty (the “hourglass phenomenon”). That hurts a generative model, which wants to spread probability over a well-used vocabulary.

So OneRec uses residual balanced K-means instead. At each level, items are clustered into K = 8192 clusters, with the clustering constrained so every cluster holds the same number of items (w = |V| / K). An item’s Semantic ID is then the sequence of nearest centroids, level by level:

Assigning a Semantic ID via residual balanced K-means (illustration; the paper gives this as Algorithm 1 pseudocode).

Each of the 3 levels has its own 8192-entry codebook, so 8192³ ≈ 5.5×10¹¹ items are representable with just 3 × 8192 codeword embeddings. A video is now 3 tokens.

Step 2 — Session-wise generation (not point-wise)#

Unlike traditional point-wise recommendation models that only predict the next video, OneRec’s session-wise generation directly generates a high-value session — a batch of ~5–10 videos returned for one user request — so the model learns both which videos to show and in what order users like them.

Training targets are high-value sessions mined from logs — e.g. sessions where the user watched ≥5 videos, total watch time exceeded a threshold, and showed interactions (like/collect/share). The model learns to reproduce these good sessions (m = 5 target items).

Step 3 — Generate the next session (encoder–decoder + MoE)#

The unified recommendation model is an encoder–decoder Transformer:

The OneRec encoder–decoder with MoE decoder (Figure 2a from the paper). Each video is 3 semantic tokens (<a_·><b_·><c_·>); the decoder emits them one at a time.

  • Encoder reads the user’s behavior sequence H_u (videos watched/liked/followed/shared, up to n = 256 of them).
  • Decoder generates the target session token-by-token (L_NTP is the Next-Token-Prediction loss), then maps each 3-token group back to a video.
  • Sparse MoE in the decoder replaces the fully-connected network (FFN) with N_MoE = 24 experts, of which a router activates only K_MoE = 2 per token. This scales model capacity without scaling inference cost — at serving, only ~13% of parameters are active. Following LLM scaling laws, the authors also found bigger OneRec models ⇒ consistently better recommendations.

Step 4 — Iterative Preference Alignment with Reward Model and DPO#

Next Token Prediction training only teaches the model to imitate good sessions. To push it toward better ones, OneRec borrows DPO (Direct Preference Optimization) from LLM post-training — but there’s a catch unique to recommendation:

In NLP, humans annotate “response A > response B” pairs. In a recommender, each browsing request only ever shows the user one list — you can never observe the same request’s “good version” and “bad version” side by side. Preference pairs don’t exist in the logs.

OneRec’s solution: train a reward model to stand in for real user feedback, then use it to improve the generator in a loop.

Reward Model#

The reward model (RM) scores how good a candidate session is — on several objectives at once. It is similar to a Multi-task Ranking model often used in a traditional recommender system. For a session S = {v₁, …, vₘ} and user u, it forms a target-aware representation of each item (eᵢ = vᵢ ⊙ u, such as target attention toward user behavior sequence), lets the items interact via self-attention, sum-pools them into a single vector, and feeds it through four Sigmoid(MLP) towers — one per objective: watch-time (swt), view-through (vtr), follow (wtr), like (ltr). It’s pre-trained on abundant logged feedback with a binary cross-entropy loss.

The reward model, based on Section 3.3.1 of the paper (which has no such figure).

The paper does not mention how an overall reward value is calculated, but it is likely a combination (e.g. weighted sum) of all predicted objectives.

Iterative Preference Alignment loop with DPO#

With the RM in hand, OneRec improves itself iteratively — generate candidate sessions, turn them into preference pairs, and train on them with DPO, then repeat with the improved model:

Iterative Preference Alignment (Figure 2b from the paper).

Generating Preference Pairs#

For each user, OneRec runs inference on the current model OneRec_t to beam-search N = 128 candidate sessions, then runs the reward model to score them. The highest-reward session becomes the chosen example and the lowest-reward one the rejected — giving a (chosen, rejected) preference pair.

Iterative Training with DPO#

Each round trains a new model OneRec_{t+1}, initialized from the current OneRec_t — which is then frozen and kept as the reference model. For every preference pair, the DPO loss pushes OneRec_{t+1} to raise the likelihood of the chosen session and lower that of the rejected one, both measured relative to the frozen reference:

Crucially, OneRec does not optimize DPO alone — it trains on the combined loss: L = L_NTP + λ·L_DPO, keeping the Next-Token-Prediction loss so the model still imitates good sessions while it learns the preference signal (pure DPO would drift away from generating valid sessions).

This runs iteratively: OneRec_{t+1} becomes the next reference model, regenerates fresh candidates and preference pairs, and the loop repeats for several rounds (OneRec_1 → OneRec_2 → … → OneRec_T) — each model bootstrapping off the previous one’s own outputs, i.e. self-improvement. To mitigate training cost, only r_DPO = 1% of training samples are trained with L_DPO loss (the rest use plain next-token loss L_NTP training). Evaluations show that a 1% DPO sample ratio achieves, on average, 95% of the maximum performance while using far less compute than higher ratios.

Experiments and Results#

Evaluated on a large-scale Kuaishou industrial dataset (offline) plus a live A/B test. Metrics: swt (session watch-time) and vtr (view-through) for watching; wtr (follow) and ltr (like) for interaction. Baselines span discriminative point-wise (SASRec, BERT4Rec, FDSA) models, generative point-wise (TIGER) models, and many DPO variants (IPO, cDPO, rDPO, CPO, simPO, S-DPO).

The high-level results:

  • Session-wise generation beats point-wise generation. OneRec-1B (without IPA) beats TIGER-1B by +1.78% max swt and +3.36% max ltr — evidence that generating a coherent list beats point predictions.
  • A tiny bit of preference alignment goes a long way. With just 1% DPO data, OneRec-1B+IPA adds another +4.04% max swt and +5.43% max ltr over base OneRec-1B, and IPA beats every other DPO variant.
  • It scales like an LLM. Growing the model from 0.05B → 1B gives consistent accuracy gains (e.g. +14.45% going 0.05B→0.1B, then further +5–6% at each step up to 1B) — the MoE design is what makes this affordable. Note that 1B is still small by LLM standards.
  • Online A/B test (Kuaishou main feed, hundreds of millions of DAU): OneRec-1B+IPA delivered +1.68% total watch-time and +6.56% average view duration — a substantial revenue-relevant lift at that scale. IPA is what pushes it over the top, and gains grow with model size:

My Thoughts#

Advantages of OneRec#

  • One model, no cascade error-stacking. Unifying retrieve+rank removes the “each stage caps the next” bottleneck and the maintenance burden of several separately-trained models. OneRec shows how such a unified generative recommendation model can be deployed at industry scale and, importantly, achieve real, substantial gains in production.
  • Session-wise is the right abstraction. Recommendation output is a list, not an item. Generating the list directly — learning content and ordering jointly — performs better than traditional point-wise recommenders.
  • MoE makes scaling practical. Getting LLM-style scaling benefits while activating only ~13% of parameters at inference is what turns “a big generative recommender” from a research toy into something Kuaishou can actually serve.
  • Self-improving alignment via DPO works for recommender systems. Recognizing that recommendation logs can’t yield preference pairs, and solving it with a self-improving loop driven by a reward model, is a beautiful innovation! It transplants LLM post-training to recommendation with real performance gains.

Disadvantages of OneRec#

  • Serving cost and latency. Beam search (size 128) over an autoregressive decoder is still far heavier than a traditional retrieve-then-rank pipeline. It needs KV-cache decoding, float16, and MoE sparsity just to be viable.
  • The reward model is now the ceiling. IPA optimizes toward RM scores, not real users. Any RM bias (e.g. over-weighting watch-time) gets amplified into the policy — a classic reward-hacking risk.
  • Added complexity. Compared to traditional recommender systems, balanced-K-means tokenization + MoE encoder-decoder + a reward model + an iterative DPO loop is a lot of moving parts added to the system.
  • Model Scaling up vs Unification. OneRec achieves substantial gains with a scaled-up unified model, but it is arguable that scaling up traditional retrieval and ranking models separately may also deliver some of the gains.

Big picture#

  • Why this paper matters. TIGER made generative retrieval competitive; OneRec is one of the first to make a single generative model beat a mature, well-tuned full cascade (retrieval + ranking) recommender system in a real large-scale production system. That’s the milestone the field was waiting for — it shows the great potential of the new Generative Recommendation direction.
  • Industry trends. This is the “recommendation is becoming an LLM problem” thesis reaching its logical conclusion: items as tokens, the pipeline as one sequence model, scaling laws + MoE for capacity, and RLHF-style preference alignment (DPO + reward model) for quality. As inference costs keep falling, expect more of the recommendation stack to collapse into such generative structures — and the self-improving alignment loop can be an exciting and key technique in recommender systems!
  • Scaling up recommender models. OneRec is one approach; there’s more to explore. OneRec-1B is still tiny by LLM standards, so as LLM inference gets cheaper, scaling the model up further is a promising path to even greater results.
