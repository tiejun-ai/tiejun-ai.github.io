    Anthropic RAG Architecture: Contextual Retrieval

    July 1, 2026

    How Anthropic architected their RAG system — Hybrid Retrieval + Contextual Retrieval + Reranking — by using an LLM at indexing time to give every chunk back the context it lost when it was split. My explanations and thoughts on such a canonical RAG architecture, and the valuable lessons for AI Engineering.

    Anthropic Engineering Blog: Introducing Contextual Retrieval

    TL;DR#

    Anthropic’s engineering blog describes their RAG architecture, built on three pieces:

    Hybrid Retrieval + Contextual Retrieval + Reranker

    The key idea is this:

    Instead of chopping a document into chunks and embedding each chunk directly, first prepend a short “context blurb” to every chunk — then index that.

    That one change, plus hybrid search and reranking, cuts the retrieval failure rate dramatically.

    1. Hybrid Retrieval: Embedding + BM25#

    The standard RAG system uses embedding semantic search for retrieval. But embeddings alone are shaky, especially for:

    • error codes
    • product model numbers
    • people’s names
    • proper nouns
    • code symbols
    • exact strings

    For example, a user asks:

    What does error code TS-999 mean?

    This query needs an exact match on TS-999. Here traditional BM25 / keyword search is often more reliable than embeddings.

    So a stronger retrieval architecture usually combines both:

    Hybrid Retrieval = Embedding Semantic Search + BM25 Text Search

    Embedding handles semantic understanding; BM25 handles exact matching. The two complement each other.

    Rank fusion. Embedding search and BM25 each return their own ranked list. Rank fusion merges the two lists into one. Common techniques:

    • Reciprocal Rank Fusion (RRF) — the most popular choice. It ignores the raw scores and uses only each document’s rank in each list, summing 1 / (k + rank) across lists (with a constant like k = 60). Because it works on ranks, it needs no score normalization and is robust across very different scoring systems. See the original paper, Reciprocal Rank Fusion outperforms Condorcet and individual Rank Learning Methods (Cormack et al., 2009).
    • Weighted score fusion — normalize each list’s scores (e.g., min-max) to a common range, then take a weighted sum (α · embedding_score + (1 − α) · BM25_score). More tunable, but sensitive to how you normalize.

    Which one does Anthropic use? The blog just says “rank fusion techniques,” but their cookbook code reveals: rank-based scoring like RRF, but with per-source weights. Each chunk’s score is weight · 1 / (rank + 1) summed over the two lists, with the semantic list weighted 0.8 and BM25 weighted 0.2 — so semantic search dominates and BM25 breaks ties and rescues exact-match cases:

    score(chunk) = 0.8 · 1/(semantic_rank + 1) + 0.2 · 1/(bm25_rank + 1)

    2. The Problem with Standard RAG: Chunks Lose Context#

    RAG normally splits a large document into many small chunks, then embeds each one:

    Embedding(chunk) = EmbeddingModel(chunk)

    The problem: once a chunk is cut out, it usually loses the context of the original document.

    Suppose a chunk is just one sentence:

    The company's revenue grew by 3%.

    But it doesn’t say:

    • Which company?
    • Which quarter?
    • Which financial report?
    • Compared to which previous quarter?

    As a result, the RAG retriever has a hard time judging whether this chunk is relevant to the user’s question.

    3. Contextual Retrieval: Give Every Chunk Its Context Back#

    Anthropic’s method: at indexing time, use an LLM to read both:

    the whole document + the current chunk

    Then generate a short 50–100 token background blurb for that chunk, and prepend it to the original chunk:

    ContextualChunk(chunk) = LLM_Context(document, chunk) + chunk

    The prompt Anthropic uses is roughly:

    <document>
{{WHOLE_DOCUMENT}}
</document>

Here is the chunk we want to situate within the whole document
<chunk>
{{CHUNK_CONTENT}}
</chunk>

Please give a short succinct context to situate this chunk within the overall
document for the purposes of improving search retrieval of the chunk.
Answer only with the succinct context and nothing else.

    Example.

    Original chunk:

    The company's revenue grew by 3%.

    LLM-generated chunk:

    This chunk is from Meta's Q2 2023 earnings report; it describes the revenue
change for that quarter relative to the previous quarter.
The company's revenue grew by 3%.

    Now the chunk itself carries far more semantic anchors:

    Meta / 2023 Q2 / earnings / revenue / QoQ growth

    …which makes it much easier for the retrieval system to recall.

    How this differs from other approaches#

    Anthropic also experimented with other ways to add context to chunks, but none worked as well as Contextual Retrieval:

    1. Adding a generic document summary to each chunk → only marginal gains in their experiments.
    2. Hypothetical Document Embedding (HyDE) → generate a hypothetical relevant document, then embed it.
    3. Summary-based indexing → build the index from summaries — lower performance in their experiments.

    The key distinction:

    Contextual Retrieval does not summarize the whole document, and does not generate a hypothetical document — it generates chunk-specific context for each chunk.

    In other words, the LLM’s job here isn’t to answer a question — it’s to improve the chunk representation during offline indexing.

    Prompt Caching: what makes this affordable#

    Running an LLM over every chunk sounds expensive — and naively it is. The context prompt includes the whole document each time (so the model can situate the chunk), and a long document may split into dozens or hundreds of chunks. Passing the full document once per chunk would mean re-reading the same document tokens over and over.

    Prompt caching removes that waste. Prompt caching optimizes your API usage by allowing resuming from specific prefixes in your prompts. This significantly reduces processing time and costs for repetitive tasks or prompts with consistent elements. As Anthropic puts it:

    With prompt caching, you don’t need to pass in the reference document for every chunk. You simply load the document into the cache once and then reference the previously cached content.

    Mechanically, you place the whole document at the front of the prompt and mark it as cacheable:

    Prompt = [ WHOLE_DOCUMENT (cached, written once) ] + [ this chunk + instruction (per call) ]

    The result is cheap enough to be practical. Using Claude 3 Haiku (a cheap and smaller model), Anthropic reports a one-time cost of $1.02 per million document tokens to generate the contextualized chunks.

    4. The Full RAG Pipeline and Architecture#

    Offline indexing stage:

    Raw document
  → split into chunks
  → generate a contextual summary for each chunk
  → context + chunk
  → build embedding index
  → build BM25 index

    Online query stage:

    User question
  → Embedding Search
  → BM25 Search
  → fuse results
  → Reranking
  → select top chunks
  → put into LLM context → generate answer

    5. Reranking Also Matters#

    Beyond Contextual Retrieval, Anthropic adds a final reranking stage at query time. It’s a two-stage retrieval pattern:

    Stage 1 — Broad recall:    hybrid search casts a wide net → e.g. top 150 candidates
Stage 2 — Precise select:  a reranker re-scores those candidates → keep top-K

    Why split retrieval in two? The cookbook puts it well:

    Initial retrieval methods (embeddings, BM25) are optimized for speed across millions of documents. Reranking models are slower but more accurate — they can afford to do deeper analysis on a smaller candidate set.

    So each stage plays to its strength:

    • First stage — retrieval is responsible for “don’t miss anything” — fast, high recall across the whole corpus.
    • Second stage — reranker is responsible for “order it precisely” — slower but sharper, over a small candidate set.

    Why bother? Two payoffs:

    1. Higher answer quality — the reranker is a far more sophisticated relevance model than the embedding similarity or BM25 scores used in stage 1, so the LLM sees the most accurate chunks with less distraction.
    2. Lower cost and latency — fewer, better chunks in the LLM’s context window means cheaper, quicker answers.

    How it works in the cookbook. Anthropic uses Cohere’s rerank-english-v3.0 model:

    1. Over-retrieve — a rule of thumb is ~10× the chunks you ultimately need (need 10 → pull 100; the blog experiments used 150).
    2. Rerank — hand the candidates plus the original query to the reranker. Crucially, each candidate is passed as both the original chunk and its generated context, giving the reranker rich signal:
    documents = [
    f"{original_content}\n\nContext: {contextualized_content}"
    for res in semantic_results
]
rerank_response = co.rerank(
    model="rerank-english-v3.0", query=query, documents=documents, top_n=k
)
    1. Select top-K — keep the highest-scoring chunks and pass only those to the LLM.

    The overhead is small: reranking adds roughly 100–200 ms and ~$0.002 per query — a modest price for the precision gain.

    6. Top-K Needs Tuning Too#

    Anthropic found that putting top-20 chunks into the LLM context worked better than top-5 or top-10.

    This matches real-world RAG experience:

    • top-K too small → easy to miss key information.
    • top-K too large → introduces noise, raises cost and latency.
    • optimal top-K must be found empirically, per task and dataset.

    In their setting, top-20 was the better choice.

    7. Final Results#

    Anthropic’s experiments show the failure rate dropping at each step:

    Baseline failure rate:                          5.7%
Contextual Embeddings:                          3.7%
Contextual Embeddings + Contextual BM25:        2.9%
+ Reranking:                                    1.9%

    In other words:

    Contextual Retrieval + Hybrid Search + Reranking significantly reduces the retrieval failure rate.

    Stacking Reranked Contextual Embedding + Contextual BM25 cuts the top-20-chunk retrieval failure rate by 67%.

    My Thoughts on Anthropic’s RAG Architecture#

    Simple yet powerful LLM-driven indexing. Using an LLM to generate contextual chunk indexes is simple yet powerful.

    Prompt caching makes it affordable. Generating a context blurb per chunk with an LLM sounds expensive, but because the whole document sits in cache for every chunk, prompt caching keeps the cost low.

    Hybrid Retrieval, Rank Fusion, and Reranking form the canonical components of a modern RAG-based AI application architecture.

    Big Picture

    An important lesson learned here is to use LLMs to improve your data representation. As inference gets cheaper, more of the classic data pipeline becomes a place to spend LLM tokens for better downstream quality.

    Anthropic’s Contextual Retrieval pioneered an effective LLM-driven approach. In newer AI agent systems such as Claude Code, traditional RAG indexing is replaced by a simple grep over files. It remains an open question how to do RAG indexing and architect RAG applications better and more effectively. More research and development will continue to improve this important area of AI engineering.

    Key Takeaways#

    • Hybrid Retrieval = Embedding (semantic) + BM25 (exact match) — the two cover each other’s blind spots.
    • The problem: chunks lose their document context, so embeddings can’t tell if they’re relevant.
    • Contextual Retrieval: at indexing time, an LLM writes a 50–100 token, chunk-specific context blurb and prepends it to each chunk — for both the embedding and BM25 indexes.
    • Reranking: a second stage that re-orders a broad candidate set (e.g., 150 → top-20) for precision, lower cost, and lower latency.
    • Results: Contextual Retrieval + Hybrid Search + Reranking drops the failure rate from 5.7% to 1.9% — a 67% reduction.
    • Core lesson: the power and magic of LLMs can help solve data-processing problems (e.g., RAG indexing) too.
