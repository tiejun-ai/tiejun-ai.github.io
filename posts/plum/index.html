    PLUM: How YouTube Adapted Gemini LLM for Industrial-scale Recommendation with Generative Retrieval — My Explanation and Thoughts

    July 18, 2026

    Understanding PLUM (Google DeepMind & YouTube) — an exciting engineering method to build an LLM as a production recommender by adapting a pre-trained Gemini LLM: tokenize every video into a Semantic ID, expand the LLM vocabulary with Semantic IDs, continue pre-training the LLM on the new YouTube video modality, then fine-tune it to generate the next video’s ID.

    Paper: PLUM: Adapting Pre-trained Language Models for Industrial-scale Generative Recommendations (He et al., Google DeepMind & YouTube, October 2025)

    TL;DR#

    PLUM is how YouTube and DeepMind built a generative retrieval LLM and deployed it into production by adapting a pre-trained Gemini LLM, serving billions of users. Three stages:

    1. Item tokenization — every video is assigned a Semantic ID (a short tuple of discrete tokens), produced by an improved RQ-VAE model the authors call SID-v2 (see my TIGER post for SID-v1).
    2. Continued pre-training (CPT) — the SID tokens are added to a Gemini LLM checkpoint’s vocabulary, and pre-training continues on a 50/50 mix of user watch histories and video metadata, so the model learns video SIDs as a new modality aligned with text.
    3. Recommendation fine-tuning — the LLM is fine-tuned to autoregressively generate the SID of the next video the user will engage with, weighted by a reward signal.

    The result: this beats a heavily optimized traditional retrieval model built on large embedding tables, while training on ~250M examples/day instead of several billion and using <0.55x the FLOPs.

    This is a significant and exciting result, as it creates an important approach to optimizing a general LLM for solving recommendation problems, and it is also one of the first generative retrieval recommendation systems launched in production at YouTube scale.

    Problem: Retrieval in Recommendation Systems#

    Input: a user’s watch history plus recommendation context.
    Output: a few hundred relevant candidate videos the user might be interested in, from a corpus of billions.

    For a decade the dominant answer has been a Large Embedding Model (LEM): learn an embedding per item (video) ID, generate an embedding for the user, and retrieve by searching with the user embedding over an ANN (Approximate Nearest Neighbor) index of item embeddings. The paper notes that in YouTube’s production retrieval model, the embedding layer uses an O(10M) vocabulary (hence O(10M) item embeddings) and holds 99.6% of the model parameters — the rest of the neural network has only 0.4%.

    The parameter budget is the whole story: LEM scales by growing lookup tables, PLUM scales by growing the network. (Diagram created for this blog, based on Section 3.1.1 of the paper.)

    That imbalance is the core limitation the paper calls out: LEMs are highly effective at memorizing user–item interactions, but concentrating the parameters in the embedding tables hinders the potential benefits of deeper, more complex networks. Its scaling methodology grows the embedding tables — in contrast to LLMs, which scale by increasing neural-network size to reason over compact tokens. You add rows, not reasoning.

    The success of LLMs has inspired an emerging industry trend of Generative Recommendation — building recommendation models with LLMs. The world knowledge and reasoning capabilities of LLMs could enable more intelligent and relevant recommendations. But adapting an LLM for recommendation is challenging.

    Why not just apply an off-the-shelf LLM directly in Recommendation Systems?#

    The primary challenge is the domain gap: an LLM is not pre-trained on user behavior data or the target item corpus, so it struggles to infer user preferences from activity and to judge nuanced item quality. As a result, directly applying an off-the-shelf LLM to recommendation shows a persistent performance gap — even on small public datasets (Hou et al., 2024).

    PLUM’s answer: Teach a pre-trained LLM a new modality — video IDs that carry semantic meaning — and domain knowledge. Represent the input (user watch history and context) as a sequence of video and text tokens. Let the LLM generate the next video IDs as recommendations based on the input tokens.

    Method Explained#

    The PLUM pipeline. (Diagram created for this blog, summarizing Section 2 of the paper.)

    Step 1 — Tokenize every video into a Semantic ID (SID-v2)#

    A Semantic ID is a tuple of discrete codewords derived from an item’s content, produced by a Residual-Quantized VAE (RQ-VAE): encode the item into a dense vector, quantize it against a codebook, take the residual, quantize that against the next codebook, and so on. Each level’s chosen codeword is one token of the ID. (For a detailed explanation of RQ-VAE, please see my TIGER post.)

    PLUM makes several changes to that recipe:

    The SID-v2 model: dual encoders, residual quantization, and reconstruction + contrastive losses. (Figure 1 from the paper, He et al., 2025.)

    • Fused multi-modal input. A video’s meaning lives in its title, its visuals, and its audio. TIGER quantized only a single (text-based) content embedding. PLUM takes a set of multi-modal embeddings {x_1 … x_M}, encodes each with its own encoder, then concatenates and projects them into one fused vector z before quantization.

    • Multi-resolution codebooks. TIGER uses the same codebook size at every level, which is inefficient: the deeper levels are encoding small, low-entropy residuals but get just as many codewords, leaving most of the SID space sparsely populated. OneRec notes a similar issue and uses a balanced K-means algorithm instead of RQ-VAE to generate SIDs (see my OneRec post for details). PLUM shrinks the codebook size for each level geometrically — 2048 / 2^(l−1) — so the first level is most discriminative and the resulting SID is more compact and efficient.

      This is paired with progressive masking, which enforces the hierarchy during training:

      semantic_id_in_training = ProgressiveMasking(semantic_id from RQ-VAE)

      Progressive masking keeps a random prefix of the SID and masks the rest, then uses only that prefix as the Semantic ID in training. For example, a 4-level SID (A5, B25, C12, D8) can be truncated to (A5, B25). This forces the earlier levels of the Semantic ID to be meaningful on their own, enforcing a stricter and more interpretable hierarchy during residual quantization. It’s similar to the Dropout technique in neural-network training: randomly dropping parts of the representation forces each part to carry useful signal on its own.

    • Co-occurrence contrastive regularization. In TIGER, the SID is a quantization of a pure content-based embedding. To better align SIDs with user behavior, PLUM injects a collaborative-filtering signal directly into the quantizer with a contrastive loss, which encourages the model to generate similar SIDs for videos that co-occur in watch sessions.

    • Three training losses. The RQ-VAE model is trained end-to-end by minimizing L = L_recon + L_rq + L_con:

      • L_con — contrastive loss: the co-occurrence regularization described above — PLUM’s addition.
      • L_recon — reconstruction loss: decodes the quantized SID back to the original input embeddings, so the SID quantization minimizes information loss.
      • L_rq — quantization (codebook + commitment) loss: trains the codebooks and the encoder to agree, so each level’s codeword faithfully represents its residual. (L_recon and L_rq are the standard RQ-VAE losses — see my TIGER post for the detailed math and explanations.)

    Step 2 — Continued Pre-Training (CPT): teach the LLM a new modality#

    A general-purpose Gemini LLM lacks YouTube’s recommendation domain knowledge — it’s a language model with its own text vocabulary. So how do you teach it about YouTube recommendation? PLUM takes a clever approach: expand the LLM’s vocabulary with video Semantic IDs.

    Just naively adding SID tokens to a pre-trained LLM’s vocabulary is not enough. We need to teach the LLM the meaning of those SID tokens by aligning them with the LLM’s existing text tokens. PLUM achieves this by continuing to pre-train the LLM with a next-token-prediction task over synthetically generated sequences of video SIDs and text tokens, on a 50/50 mixture:

    • User behavior data — watch histories, with extra features.
    • Video metadata corpus — natural-language statements that bind a SID to its text, drawn from the video’s title, description, captions, and channel name.

    Both are serialized into token sequences using the following schemas:

    SchemaExample
    User behaviorwatch_history = <sid_1> <channel_name> <watch_ratio> <watch_time> <hours_since_final_watch> <sid_2> <channel_name> … <sid_n>
    SID + video titleVideo <sid> has title (en): <video_title>
    SID + video topicsThe topics in video <sid> are: <topics>

    Example schemas used in continued pre-training. (Table 1 from the paper.)

    Training Scale: 1 million steps at batch size 16, roughly 260 billion tokens.

    The result is an LLM where SID tokens live in the same representational space as text — and, notably, it retains general LLM ability and can do few-shot in-context learning over SIDs. Given a SID and a task description, it completes the phrase sensibly.

    Step 3 — Supervised Fine-Tuning (SFT) for generative retrieval#

    CPT gives you a model that understands SIDs. SFT specializes it for retrieval: given a user and context, generate the SID of the video they will engage with next.

    Generative retrieval: the decoder-only LLM reads the prompt and generates the next video’s Semantic ID. (Figure 2 from the paper, He et al., 2025.)

    The input prompt is a mix of modalities — interleaved SID tokens, text features, and custom tokens for numerical features. In particular, real-time context is added, which is missing in CPT. The model learns to predict the SID tokens of clicked videos from user logs given user context and history.

    The loss is a standard autoregressive maximum-likelihood objective over the target SID’s tokens, but weighted by a reward:

    r(user, v_click) is a handcrafted reward for each click — so a click that led to a satisfying watch counts for more than an accidental one. Training examples are sampled by this reward weight and then weighted equally in the loss.

    At serving time, PLUM uses beam search to decode multiple SIDs from the model; each decoded SID maps back to a real video as a retrieval candidate. In principle the model can hallucinate a SID that maps to no video, but after SFT the hallucination rate is <5% — a real but manageable problem.

    And there is no ANN index: the model is the index.

    Experiments and Results#

    The evaluated model is a 900M activated-parameter Mixture-of-Experts (MoE) (~4.2B total) LLM from the Gemini-1.5 family, warm-started, continually pre-trained and fine-tuned on YouTube data, for both long-form video (LFV) and Shorts.

    Against a highly optimized production Transformer-based LEM baseline (each number is PLUM ÷ LEM):

    Effective vocab size2.60x13.24x
    CTR1.42x1.33x
    Watch time / view0.72x1.13x
    Watch fraction / view1.32x1.03x

    Table 2 from the paper. “Effective vocab size” = number of unique videos needed to cover 95% of impressions.

    The standout is effective vocab size — 13x on Shorts. PLUM recommends a dramatically wider slice of the video corpus, which means the PLUM model generalizes better, whereas baseline LEM concentrates far more on popular videos. CTR is up meaningfully in both surfaces.

    Live A/B tests, adding PLUM’s retrieval on top of the best existing retrieval model (quota-matched, denoted LEM+):

    MetricLFVShorts
    Engaged users+0.07%+0.28%
    Panel CTR+0.76%+4.96%
    Views+0.80%+0.39%
    Satisfaction+0.06%+0.39%

    Table 3 from the paper.

    These improvements look small in absolute terms; at YouTube scale they are not. They show that there is additive value on top of a highly optimized system.

    Training efficiency. The 900M MoE trains on roughly 250M examples/day; the LEM trains on several billion per day. Despite having 100x more dense parameters than LEM, PLUM’s total training cost is comparable — it converges much faster, using <0.55x the FLOPs.

    Do CPT and initialization with a pre-trained LLM actually help?

    Recall that:

    PLUM training = Init-with-Pre-trained-LLM + Continued-Pre-Training (CPT) + Supervised-Fine-Tuning (SFT)

    Evaluation results for 4 variants, all on the same MoE-900M architecture, measured by 8th-day Recall@10:

    ModelPre-trained LLMCPTRecall@10
    R1NoNo0.19
    R2YesNo0.23
    CR1NoYes0.27
    CR2 (PLUM)YesYes0.28

    Table 5 from the paper.

    Read this carefully — it is the paper’s most useful and interesting table:

    • Benefits of CPT. There is a large performance gap between R1 and CR1 (or R2 and CR2). Furthermore, a single CPT checkpoint can be reused across many downstream fine-tuning tasks.
    • Benefits of Pre-trained LLM. Models initialized from a pre-trained LLM consistently outperform their randomly initialized counterparts with or without CPT. The advantage may come from the pre-trained LLM’s knowledge of natural languages.
    • Diminishing returns when combined (CR1 → CR2). Initializing from the pre-trained LLM when CPT is applied (CR1 → CR2) lifts Recall@10 by only +0.01 (0.27 → 0.28), versus +0.04 when there is no CPT (R1 → R2). Once CPT has adapted the model to domain data, the pre-trained weights contribute much less — CPT already captures most of what the LLM initialization would provide. This is an interesting observation as we might expect the general knowledge in the LLM to help more. One factor could be that the LLM used is still relatively small.

    Scaling study. Across MoE-110M / 370M / 900M / 3B, training loss follows a clean power law in Iso-FLOPS, and retrieval Recall@10 keeps improving with compute. Optimal model size shifts larger as the compute budget grows.

    One caveat, which the authors flag: MoE-3B did not beat MoE-900M at the training budgets tested. They attribute this to their training setup — every model got the same training resources with batch size saturating HBM (High Bandwidth Memory), which assigns a smaller batch size to larger models, and the MoE-3B model was trained with only 0.57 epochs of data. The lesson they draw is that compute-optimal training requires scaling training data and model size together.

    My Thoughts#

    Advantages and Lessons#

    • Adapting an LLM for recommendation. PLUM introduces an effective framework to adapt a pre-trained LLM for generative recommendation. The structure — tokenize, continue pre-training, fine-tune — is a genuinely transferable playbook for bringing an LLM into any multi-modality domain for domain-specific tasks, beyond generative recommendation.

    • Better Semantic IDs from RQ-VAE. Several tokenizer innovations drive higher-quality Semantic IDs: a co-occurrence contrastive loss that injects collaborative-filtering signal, multi-resolution codebooks, and progressive masking. Good Semantic IDs are foundational to any generative recommendation system.

    • Training efficiency changes the economics. Needing ~20x fewer training examples per day is what makes an LLM affordable for recommendations here at all. This is one strong argument that item-token architectures are cheaper than large embedding tables.

    • A fully deployed LLM recommender. PLUM turns a pre-trained Gemini LLM into a production recommendation LLM, deployed at YouTube scale. As one of the first LLMs deployed in production for generative recommendation, this is an exciting and significant engineering achievement.

    Disadvantages and Possible Future Work#

    • Serving and infra cost. Not specific to PLUM, but the serving cost of LLM inference plus beam search may still be higher than a traditional LEM model with ANN search.

    • Watch time per view drops on LFV (0.72x). Retrieving more diverse, niche videos seems to come at the cost of shorter watches on long-form videos. One possible remedy is to assign a higher watch-time weight in the SFT reward and sampling.

    • Memory-efficient SFT (LoRA / QLoRA). PLUM uses full-parameter fine-tuning for SFT. It would be worth studying whether memory-efficient techniques (LoRA, QLoRA) offer a better trade-off between training cost and model performance.

    • Scaling law and bigger LLMs. The MoE-3B model failed to beat MoE-900M. Further work might resolve this scalability issue and enable larger LLMs for recommendation — potentially bringing more of the “LLM magic” to generative recommendation.

    Big picture#

    Why it matters. As LLMs grow more capable — absorbing multi-modal knowledge and reasoning ever better — applying them to recommendation holds great potential. PLUM is one of the first systems to successfully adapt a pre-trained, general-purpose LLM into a recommendation LLM for generative recommendation, and to fully deploy it at YouTube scale. That is an exciting and remarkable engineering achievement.

    More importantly, PLUM offers a repeatable methodology — tokenize, continue pre-training, fine-tune — for turning a general LLM into a specialized LLM for recommendation. That recipe might also be an important contribution to domain-specific LLM applications, beyond generative recommendation.

    Industry trend. LLM- and Scaling-Law-driven generative recommendation is now an important industry direction. Pioneering systems like TIGER, OneRec, and PLUM provide good foundations for understanding it and building further. And as LLMs keep getting more powerful and more efficient, it is exciting to watch the development of generative recommendation!
